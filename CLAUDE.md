# 1MoreRep — contributor & agent guide

Self-hosted gym-tracker PWA. Next.js 15 (App Router, RSC + server actions),
React 19, Prisma + PostgreSQL, next-intl (en/de), deployed via Docker Compose.

## THE GOLDEN RULE: `./update.sh` must upgrade ANY prior version, every time

The **only** thing an operator ever runs to upgrade a live instance is:

```bash
./update.sh          # pull → rebuild → migrate → seed → health-check
```

It must take an instance running **any previously released commit** to the latest
one with **zero manual steps** — no hand-run SQL, no "first do X then upgrade",
no data loss, secrets preserved. Treat this as a hard contract on every change.

This works because of three layers; **every change you make must keep all three intact**:

1. **Forward-only Prisma migrations.** Boot runs `prisma migrate deploy`
   (see `docker-entrypoint.sh`), which applies *every* unapplied migration in
   order, regardless of how old the starting DB is. So:
   - **Never edit or delete an already-committed migration** in
     `prisma/migrations/`. Add a *new* one. Editing history breaks every
     instance that already applied the old version.
   - Any schema change (`prisma/schema.prisma`) MUST ship a matching migration
     generated with `pnpm prisma migrate dev --name <change>`. A schema edit
     without a migration = broken upgrade.
   - Write migrations to be **safe on populated tables**: new columns are
     nullable or have defaults; renames/backfills are done as additive steps
     (add new → backfill → switch reads/writes → drop old in a *later* release),
     never a destructive in-place change that loses data.
   - Migrations must be **idempotent-safe to re-run** is guaranteed by Prisma's
     `_prisma_migrations` ledger — don't defeat it with manual DDL.

2. **Idempotent seed.** Boot also runs `prisma/seed/index.ts`. It must be safe to
   run on every container start against a fully-populated DB — `upsert`/`skip`,
   never blind `create`. New seed data (exercises, settings, defaults) is
   backfilled here so old instances gain it on upgrade. Adding seed content is
   the supported way to ship new default data to existing installs.

3. **`update.sh` itself stays self-contained & self-updating.** It re-execs the
   freshly-pulled copy if it changed (so new deploy logic runs), preserves
   `.env`, and health-checks `/api/health`. If you change the deploy/runtime
   shape (new env var, new service, new build step), update `update.sh`,
   `install.sh`, `docker-compose.yml`, `Dockerfile`, and `.env.example`
   **together**, and make new env vars optional with a sane default so an old
   `.env` keeps working.

**Before finishing any change that touches the DB, deploy, or runtime, ask:**
"If someone runs `./update.sh` on a 3-versions-old instance with real data,
does it come up clean with their data intact and no manual steps?" If not, fix
it before shipping.

## Dev commands

```bash
pnpm dev                       # local dev server
pnpm typecheck                 # tsc --noEmit
pnpm test                      # vitest (unit/domain)
pnpm e2e                       # playwright
pnpm prisma migrate dev --name <change>   # schema change → new migration
pnpm build                     # production build
```

## Conventions

- **i18n:** every user-facing string goes through `next-intl`; add keys to BOTH
  `messages/en.json` and `messages/de.json`. Locale is **account-bound**
  (`User.locale`), not browser-local. Server channels (email/push/cron) render in
  the recipient's locale via `getTranslator(user.locale)`.
- **Secrets at rest:** AES-256-GCM via `src/lib/crypto.ts` (`encryptSecret`/
  `decryptSecret`), keyed by `APP_KEY`. Never store secrets in plaintext columns.
- **Auth/session:** opaque DB sessions (`src/lib/auth/session.ts`); server
  actions are the only mutation entry points and must re-derive the actor from
  the session, never trust a client-supplied user id.
- The muscle taxonomy in `src/domain/muscles/taxonomy.ts` mirrors the Prisma
  `Muscle` enum — keep them in sync.

## Secrets & public-repo hygiene — nothing sensitive reaches GitHub

`origin` is a **public** GitHub repo, so **every pushed branch is world-readable**
— there is no "private branch" here. Keep the entire push surface secret-free, in
new files *and* in history.

- **No real secrets, ever.** The only committed env file is **`.env.example`**
  (placeholders only). Real `APP_KEY` / VAPID keys / SMTP creds / the production
  DB password are generated by `install.sh` into `.env` (git-ignored) or stored
  AES-256-GCM-encrypted in the DB — never in a tracked file.
- **`devpassword` is NOT a secret** — it's the documented local/CI Postgres
  default (`docker-compose.yml`, `.github/workflows/ci.yml`, `playwright.config.ts`,
  `src/lib/env.ts`); production overrides it via `.env`. Don't "fix" it with a real
  password in a tracked file — *that* would be the actual leak.
- **`.gitignore` is load-bearing**: it keeps `.env*` (except `.env.example`),
  `/data` (uploads), `/.claude/`, `/.analysis/`, and `tests/e2e/.auth/` out of the
  repo. Add any new secret-adjacent path to it in the same change.
- **Keep internal/sensitive material off the repo entirely** — if it can't be
  public, it doesn't get committed or pushed on any branch.
- **Commit identity:** public commits use the GitHub **noreply** email, never a
  personal/company work email.
- **Run the leak check before every push** (covers new files *and* history):

  ```bash
  gitleaks detect --source . --log-opts=main   # public history (HEAD/main)
  gitleaks detect --source . --no-git          # working tree, incl. untracked
  git log --all --format='%ae %ce' | sort -u   # no work email on pushed branches
  ```
